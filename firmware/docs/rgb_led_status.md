# BBClaw 三色状态灯设计（绿=就绪，黄=处理中，红=问题）

## 目标

这版状态灯改成更严格的三色语义：

- **绿灯**只表示“设备已就绪，可继续用”
- **黄灯**表示“后台有链路在处理”
- **红灯**只在“遇到问题/阻塞异常”时出现

同时满足两个约束：

- **任一时刻只表达一个主状态**
- **用户能看出当前是谁在处理消息**

这里的“谁”统一分成三层：

1. **设备本机**：按键、录音、封包、VAD、本地展示
2. **adapter**：本地 HTTP / ASR / TTS / 调度层
3. **上游 OpenClaw**：Gateway / `nodes` / 远端 agent

## 总原则

### 1. 颜色只表达大类

- **绿**：稳定、可继续操作
- **黄**：处理中，但系统仍然活着
- **红**：异常，需要等待恢复或人工关注

### 2. 谁在处理，用“节奏”表达

黄灯不再靠换颜色区分阶段，而是靠**闪烁节奏**区分处理归属：

- **黄常亮**：设备本机正在处理
- **黄单闪循环**：消息正在路上，等待 adapter 接手
- **黄双闪循环**：adapter 正在处理
- **黄三闪循环**：上游 OpenClaw / agent 正在处理

处理完成后，回到绿灯前再给一次**短确认脉冲**：

- **绿 1 脉冲后常亮**：本机闭环完成
- **绿 2 脉冲后常亮**：adapter 闭环完成
- **绿 3 脉冲后常亮**：上游 OpenClaw / agent 闭环完成

这样用户看一眼最后一次“回绿”动作，就知道这一轮最终落到了哪一层。

## 主状态表

| 状态 | 灯效 | 含义 | 备注 |
|------|------|------|------|
| 冷启动 / 唤醒恢复 | 跑马灯 | 系统正在起来 | 仅启动期出现 |
| 等待 Wi‑Fi / adapter | 黄单闪循环 | 基础链路尚未 ready | 还不能算就绪 |
| 已连上 adapter，可工作 | 绿常亮 | Ready | 稳定默认态 |
| 按住说话 / 本地采音 | 黄常亮 | 当前由设备本机处理 | 用户动作已被接住 |
| 松开后本地收尾 | 黄常亮转黄单闪 | 设备封包后开始送 adapter | 本机处理转入链路传递 |
| adapter 本地处理 | 黄双闪循环 | adapter 在处理消息 | 例如本地 ASR / TTS / 规则 |
| 上游 OpenClaw 处理 | 黄三闪循环 | 消息已进入 Gateway / `nodes` / agent | 保留给上游链路 |
| 完成并回就绪 | 绿脉冲后绿常亮 | 本轮已完成 | 脉冲次数表示最终处理方 |
| 可恢复异常 | 红慢闪 | 出问题，但仍在重试 | 例如 heartbeat 短暂失败 |
| 阻塞异常 | 红双闪循环 | 当前流程失败或不可用 | 例如音频失败、adapter 长时间不可达 |

## 具体灯效参数

为了让肉眼容易区分，节奏尽量简单，避免太花。

### 启动跑马灯

- 顺序：**红 -> 黄 -> 绿**
- 每步：`120 ms`
- 冷启动：`2` 圈
- 从休眠恢复 / 快速重连：`1` 圈

说明：

- 终点必须落在**绿常亮**或**黄单闪**，不能停在红灯
- 如果启动后 adapter 还没 ready，跑马灯结束后立刻切到**黄单闪**

### 就绪

- **绿常亮**

这是唯一长期稳定态。  
只要 adapter heartbeat 正常、设备可接收输入，就应回到这个状态。

### 黄灯节奏

#### 1. 本机处理

- **黄常亮**

用于：

- PTT 按下后正在录音
- 松开前仍在本地采集
- 本地 VAD / 封包 / UI 同步

#### 2. 路由中，等待 adapter 接手

- **黄单闪循环**
- 节奏：`200 ms 亮 + 800 ms 灭`

用于：

- 已结束本地采音，准备把消息交给 adapter
- 启动阶段等待 adapter health ready

#### 3. adapter 处理中

- **黄双闪循环**
- 节奏：`120 ms 亮 + 120 ms 灭 + 120 ms 亮 + 840 ms 灭`

用于：

- adapter 本地 ASR
- adapter 本地规则处理
- adapter 本地 TTS 合成
- adapter 已接住请求，但还没交给上游

#### 4. 上游 OpenClaw / agent 处理中

- **黄三闪循环**
- 节奏：`120 ms 亮 + 120 ms 灭 + 120 ms 亮 + 120 ms 灭 + 120 ms 亮 + 520 ms 灭`

用于：

- adapter 已把消息送入 Gateway
- Gateway / `node.event` / `node.invoke` / agent 正在处理
- 这部分是后续 `nodes` 主线接入后要重点保留的语义

### 回绿确认

处理完成时，不直接硬切绿常亮，而是先打一小段绿脉冲：

| 完成归属 | 灯效 |
|------|------|
| 本机完成 | 绿 1 脉冲，各 `160 ms` |
| adapter 完成 | 绿 2 脉冲，各 `160 ms` |
| 上游完成 | 绿 3 脉冲，各 `160 ms` |

脉冲结束后统一回到：

- **绿常亮**

### 红灯异常

#### 1. 可恢复异常

- **红慢闪**
- 节奏：`250 ms 亮 + 1750 ms 灭`

用于：

- adapter heartbeat 首次失败
- 临时网络抖动
- 正在后台重连，但还没确认彻底失败

#### 2. 阻塞异常

- **红双闪循环**
- 节奏：`150 ms 亮 + 150 ms 灭 + 150 ms 亮 + 1050 ms 灭`

用于：

- 音频初始化失败
- stream finish 失败
- adapter 长时间不可达
- 上游返回明确错误，导致本轮无法继续

红灯恢复条件：

- 一旦健康检查恢复，先给一次**绿 2 脉冲**，再回**绿常亮**
- 红灯不和任何黄/绿叠加

## 推荐状态流

### 1. 开机到 ready

`跑马灯 -> 黄单闪(等 Wi‑Fi / adapter) -> 绿常亮`

解释：

- 是否“ready”，以 **adapter 已可用** 为准
- 只连上 Wi‑Fi 但 adapter 未 ready，不算 ready

### 2. 一轮本地对讲

`绿常亮 -> 黄常亮 -> 黄单闪 -> 黄双闪 -> 绿 2 脉冲 -> 绿常亮`

含义：

- 用户按下后，本机接住录音
- 松开后，消息交给 adapter
- adapter 本地完成处理并返回

### 3. 一轮上游 agent 处理

`绿常亮 -> 黄常亮 -> 黄单闪 -> 黄双闪 -> 黄三闪 -> 绿 3 脉冲 -> 绿常亮`

含义：

- adapter 接到消息后又交给了 OpenClaw `nodes` / agent
- 最终由上游完成，所以回绿时打 3 脉冲

### 4. 异常恢复

`绿常亮 -> 红慢闪/红双闪 -> 绿 2 脉冲 -> 绿常亮`

## 与当前固件枚举的映射建议

当前 `firmware/include/bb_led.h` 里的枚举还带有旧语义，需要逐步收敛：

| 现枚举 | 新语义建议 |
|------|------|
| `BB_LED_IDLE` | 绿常亮 |
| `BB_LED_RECORDING` | 黄常亮 |
| `BB_LED_PROCESSING` | 拆成更细状态，不再代表所有“处理中” |
| `BB_LED_REPLY` | 改成“回绿确认脉冲”，并带处理归属 |
| `BB_LED_NOTIFICATION` | 不建议继续单独占颜色；如果保留，应并入黄灯节奏 |
| `BB_LED_SUCCESS` | 并入“回绿确认脉冲” |
| `BB_LED_ERROR` | 红慢闪或红双闪，按错误级别分 |

建议后续把状态枚举改成更接近链路归属的命名，例如：

- `BB_LED_READY`
- `BB_LED_LOCAL_ACTIVE`
- `BB_LED_ROUTE_TO_ADAPTER`
- `BB_LED_ADAPTER_ACTIVE`
- `BB_LED_UPSTREAM_ACTIVE`
- `BB_LED_DONE_LOCAL`
- `BB_LED_DONE_ADAPTER`
- `BB_LED_DONE_UPSTREAM`
- `BB_LED_WARN`
- `BB_LED_ERROR`

## 硬件实现说明

当前硬件按 **R / Y / G 三路独立引脚**实现：

- **业务语义上**仍然是“单色互斥”
- **驱动实现上**黄灯直接使用独立 `Y` 引脚
- 上层状态机不能把多个颜色叠加理解成多个并发主状态

也就是说，“不能同时点亮”在语义上理解为：

- **同一时刻只能有一个主灯语**
- 不是同时表达“就绪”和“错误”

## 当前落地建议

先按下面顺序改，风险最小：

1. 保留开机跑马灯，但顺序改成 **红 -> 黄 -> 绿**
2. `adapter ready` 前统一显示 **黄单闪**
3. `adapter ready` 后统一回 **绿常亮**
4. 把“录音 / VAD / 上传 / adapter 处理 / 上游处理”全部收进黄灯体系
5. 把红灯收窄到真正异常
6. 最后再把“处理归属”接入绿脉冲确认

## 结论

这套方案的核心是：

- **颜色只看大类**
- **节奏才看归属**

这样不会把三色灯做成“颜色过载”的状态表，用户也能知道：

- 现在是不是 ready
- 现在是不是在忙
- 忙的是设备、adapter，还是上游 OpenClaw
- 这轮最终是谁处理完的
