# BBClaw Protocol Specs

## Voice Verify

Voiceprint unlock uses the existing WebSocket envelope and adds a dedicated verification exchange before normal voice sessions are allowed.

### Device -> Cloud

```json
{
  "type": "request",
  "messageId": "verify-1712736000000",
  "deviceId": "BBClaw-0.1.0-A1B2C3",
  "kind": "voice.verify",
  "sessionKey": "agent:main:bbclaw-a1b2c3",
  "sampleRate": 16000,
  "channels": 1,
  "audioBase64": "<base64 pcm16 mono>"
}
```

Fields:

- `audioBase64`: PCM16 little-endian, 16kHz, mono
- `messageId`: per-request identifier generated by firmware
- `sessionKey`: same device session identity used by voice streaming

### Cloud -> Device

```json
{
  "type": "event",
  "kind": "voice.verify.result",
  "match": true,
  "confidence": 0.95,
  "message": "stub mode"
}
```

Fields:

- `match`: whether the passphrase check succeeded (ASR transcript vs expected phrase; not biometric voiceprint)
- `confidence`: float confidence score from the verifier
- `message`: human-readable status or failure reason

## Device State

- `LOCKED`: default state for `cloud_saas` transport after boot completes. PTT captures PCM locally and sends `voice.verify` instead of starting a normal voice session.
- `UNLOCKED`: entered only after `voice.verify.result.match == true`. PTT resumes the existing `voice.stream.start` / `voice.stream.finish` flow.
- Pairing, auth, WiFi, and cloud-link states still take precedence on screen so the user can finish setup before unlock becomes usable.

## Cloud

- WebSocket: consume `kind="voice.verify"` and emit `kind="voice.verify.result"` (see `bbclaw-reference` cloud).
- HTTP (optional / same semantics): `POST /v1/voiceprint/verify` — path name is legacy; verification is **passphrase via ASR**, not speaker biometrics.
